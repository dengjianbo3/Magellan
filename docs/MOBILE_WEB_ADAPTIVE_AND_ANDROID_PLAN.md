# 移动端自适应 + 安卓 App 化改造规划

## 1. 目标与原则

### 1.1 目标
- 在手机浏览器中获得完整可用、沉浸式、低学习成本的体验。
- 在不重写前端框架的前提下，产出可安装的安卓 App。
- 保持当前“原子 Agent + 编排”能力不变，优先改造呈现层与容器层。

### 1.2 设计原则
- 同一套业务逻辑，多端复用（Web/Android 共用 Vue 代码）。
- 先可用再增强：先完成移动端自适应，再做 App 封装与原生增强。
- 以性能和稳定性优先，避免“看起来像 App 但不好用”。

## 2. 当前技术基线（已确认）
- 前端：Vue 3 + Vite + Vue Router + Pinia + Tailwind。
- 路由：`createWebHistory`，已具备 SPA 结构。
- 现状适合采用“响应式 Web + Capacitor 安卓容器”方案。

## 3. 总体方案（推荐）

## 3.1 两阶段架构
1. 阶段 A：移动端 Web 自适应（手机浏览器可直接用）
2. 阶段 B：安卓 App 封装（Capacitor）

### 3.2 为什么不直接原生重写
- 成本高、周期长、与现有迭代节奏冲突。
- 当前核心痛点在布局交互与端侧容器，不在业务逻辑层。

## 3.3 安卓技术路线选择
- 推荐：**Capacitor**
- 备选：TWA（更快但能力受限）、纯 WebView 壳（工程质量较弱）

选择 Capacitor 的理由：
- 与 Vue/Vite 兼容好，改造成本低。
- 后续可平滑接入原生能力（剪贴板、通知、文件权限、分享）。
- 比纯 TWA 更适合你当前“长会话 + WebSocket + 多模态输入”场景。

## 4. 阶段 A：移动端自适应改造

### 4.1 布局策略
- 断点建议：
  - `>=1200`: 桌面（保持现有双侧栏体验）
  - `768-1199`: 平板（侧栏可收起）
  - `<768`: 手机（单栏主内容 + 抽屉/底部导航）
- 手机端布局：
  - 顶层仅保留主内容区；
  - 历史会话、专家列表改为抽屉（左/右滑出）；
  - 输入框固定底部，考虑安全区（刘海屏）。

### 4.2 关键页面改造优先级
1. 专家群聊页（ChatHub / AgentChat）
2. 头脑风暴页（Roundtable）
3. 分析页（Analysis）
4. 知识库页（Knowledge）
5. 设置页（Settings）

### 4.3 交互与可用性细节
- 输入法回车与“发送”分离（中文输入法组合态不触发发送）。
- 键盘弹起时输入框和消息区自适应，不遮挡最近消息。
- 自动滚动到底逻辑在移动端单独校准。
- 历史会话折叠/展开动画简化，减少卡顿。

### 4.4 样式与组件层改造
- 抽离全局尺寸与间距 token（移动端紧凑模式）。
- 卡片/列表/表格统一移动端变体（表格转卡片）。
- Markdown 渲染样式增加移动端 typography 规则（行长、代码块横滑）。

### 4.5 性能优化（移动端优先）
- 首屏 JS 分包再细化（路由级 lazy load）。
- 图片与图表按需渲染，非首屏组件延迟加载。
- 长列表虚拟化（会话记录、消息流）。

## 5. 阶段 B：安卓 App 化（Capacitor）

### 5.1 App 封装步骤
1. 前端构建产物稳定（`frontend/dist`）。
2. 初始化 Capacitor 工程并接入 Android 平台。
3. 配置 App ID、应用名、图标、启动图、权限。
4. 接入网络配置（线上 API 域名、HTTPS、WebSocket）。
5. 打包 Debug APK 与 Release AAB。

### 5.2 运行时能力规划
- 第一期（必须）：
  - 登录、会话、WebSocket 实时消息、图片上传。
- 第二期（增强）：
  - 剪贴板图片粘贴（原生插件增强可靠性）。
  - 推送通知（分析完成、讨论完成提醒）。
  - 文件分享到 App（从系统分享面板打开）。

### 5.3 发布流程
- 生成 keystore，配置签名。
- 产出 release AAB，上传 Google Play 内测轨道。
- 建立版本号策略：`major.minor.patch+build`.

## 6. 里程碑与交付

### M1（移动端可用基线）
- 完成核心页面手机自适应。
- 输入框、滚动、侧栏抽屉、历史会话操作稳定。
- 交付：移动端 UI 基线 + 自测报告。

### M2（移动端体验优化）
- 完成性能优化和渲染细节统一。
- 交付：性能对比数据（首屏、交互响应、长会话流畅度）。

### M3（安卓 App 首版）
- 完成 Capacitor 安卓封装，产出可安装 APK。
- 交付：APK + 安装说明 + 已知问题列表。

### M4（安卓发布版）
- 完成签名与 AAB 打包，具备上架条件。
- 交付：Release AAB + 发布清单。

## 7. 验收标准

### 7.1 手机浏览器验收
- 主流程可完整闭环：登录 -> 提问 -> 专家响应 -> 历史查看。
- 页面无横向溢出，无关键按钮被遮挡。
- 软键盘弹起后输入与发送行为正确。
- 长会话滚动流畅，无明显掉帧阻塞。

### 7.2 安卓 App 验收
- 可安装、可登录、可持续会话。
- WebSocket 在前后台切换后可恢复。
- 图片上传和多模态提问可用。
- 崩溃率和白屏率可控（内测阶段）。

## 8. 风险与应对
- 风险：移动端双侧栏信息密度过高。
  - 应对：手机端改为单栏 + 抽屉，不强行保留桌面结构。
- 风险：WebSocket 在弱网/切后台断连。
  - 应对：统一重连策略 + 会话恢复机制 + 心跳。
- 风险：安卓机型碎片化导致兼容问题。
  - 应对：建立最小设备矩阵（小米/华为/三星/Pixel）。
- 风险：多模态上传在 WebView 行为差异。
  - 应对：优先使用 Capacitor 官方文件与相机插件。

## 9. 建议执行顺序
1. 先完成 M1（手机浏览器可用），避免“先打包后返工”。
2. 再做 M2（流畅度和细节）。
3. 最后推进 M3/M4（安卓封装与发布）。
